# Self-Review

직전 답변(또는 지정한 대상)을 "처음 보는 사람이 트집 잡는다"는 자세로 다시 읽고, **수정점만** 보고한다.
목적은 사용자가 매번 손으로 "정말 확실해?"를 묻는 라운드를 대신하는 것이다. 칭찬·요약·반복은 빼고, 실제로 바뀌어야 할 것만 낸다.

## 무엇이 '더 나은 답'인가 — 근거(grounding)가 유일한 기준

답이 *바뀌었다고* 나아진 게 아니다. 모델이 "이제 확실하다"고 *말하는* 것도 증거가 아니다. 믿을 수 있는 수정은 **첫 답이 안 본 증거를 가져왔을 때뿐이다.** 그래서:

- **[수정]은 증거가 붙을 때만 한다.** "X가 틀렸다 → Y (근거: 파일 △△ 27줄 / 실행 결과 / 문서 인용 / 구체적 반례)". **증거를 못 대면 고치지 않는다.**
- 본질적으로 검증 불가능한 것(판단·예측·취향)은 → **[불확실]로 표시만 하고 답은 그대로 둔다.** 추측으로, "responsive하게 보이려고" 멀쩡한 답을 바꾸지 마라(아첨 금지).
- 한계 인정: 같은 모델은 증거를 *날조·오독*할 수 있다. 그래서 증거 요구는 아첨을 *불가능*하게 만드는 게 아니라 *문턱을 크게 높일* 뿐이다 — 이 약점을 백지 컨텍스트의 독립 시선으로 일부 보완한다(아래).

## 검토 방식 — 단일 모드, 필요시 자동 격상

모드 선택은 없다. 그냥 `/self-review`. 스킬이 스스로 판단한다.

**먼저 무엇을 검토할지 정한다** — 인자가 *초점 영역*(보안/성능 등)이거나 비어 있으면 **직전 답변**을 검토한다. 인자가 *대상*(예: "X 파일", "방금 만든 PR")이면 직전 답변 대신 **그 대상**을 검토한다.

- 기본: 프레임을 바꿔 대상을 다시 읽는다(빠름).
- **검증할 거리가 있거나 스테이크가 높으면**(실행될 코드, 되돌리기 어려운 작업, 외부로 나가는 산출물) 도구로 직접 확인하고, **독립 에이전트를 1개 띄워 백지에서** 같은 대상을 검토하게 한다(질문 원문+검토 대상 전달). 주의: 이 에이전트는 기본적으로 *같은 모델*이라 모델이 달라서 생기는 decorrelation은 없다 — 상관 감소는 오로지 백지 프레임(앵커링 제거)에서 온다. 그래서 같은 모델 블라인드스팟은 *일부만* 깨진다.

## 검토 축

- **사실/근거** — 도구로 확인 안 하고 기억에만 의존한 주장은? → 지금 도구로 확인.
- **가정** — 조용히 깐 전제는? 틀리면 답이 무너지는 곳은?
- **누락** — 빠뜨린 케이스·엣지·요구사항·예외는?
- **오판** — 잘못 읽었거나 논리 비약한 곳은?
- **대안** — 더 나은 방법을 안 보고 넘어갔나?
- **범위** — 요청을 너무 좁게/넓게 해석했나?

> 이 축은 전역 규칙 11의 자기검증 축과 거의 동일하다(여기에 '대안'이 추가됨). 규칙 11=항상-로드 인라인 경량 패스, 이 스킬=같은 축의 온디맨드 심층 패스. 축을 바꿀 땐 양쪽을 함께 본다(drift 방지).

코드/명령/설정이 답에 포함됐으면 가능한 범위에서 실제로 돌려보거나 파일을 다시 읽어 확인한다(기억으로 단정 금지).

## 종료 조건 (churn 방지)

- **라운드 시작 시 검토할 주장 목록을 고정(freeze)** 한다 — 검증하다 떠오른 잔가지로 무한히 번지지 않게.
- 한 라운드가 *실질* 수정(정확성·누락·오판)을 못 내면 멈춘다. 표현·스타일 트집은 수정으로 치지 않는다.
- **최대 2라운드** 백스톱. (1라운드: 검토→수정. 2라운드: 그 수정이 새 문제를 만들었는지/핵심과 관련 있는지 정합성 확인 — 아래 '변경 후 정합성 확인'을 별도 패스로 적용.) 독립 에이전트 격상은 1라운드 *내부* 절차로 친다(별도 라운드로 세지 않는다). 그 이상은 거의 헛돎이라 자른다.

## 변경 후 정합성 확인

증거 있는 수정이라도, 반영 뒤 **① 그게 질문의 핵심과 관련 있나 ② 답의 다른 부분·인접 코드와 부딪치지 않나**를 한 번 본다. (국소적으로 옳지만 전역을 깨는 수정 방지.)

## 적용 방식 — 검토는 실행이 아니다

전역 규칙 12(계획·검토 요청 시 실행 금지)를 따른다. 검토 대상이 **아직 실행 안 된 계획·제안**이면 수정을 *제안*으로만 내고 직접 안 고친다(승인 게이트 유지). **이미 적용된 산출물**이면 고치되 무엇을 왜 고쳤는지 보고한다. 애매하면 묻는다.

## 출력 형식

걸린 항목만, 라벨 붙여 짧게:

```md
[수정] {무엇이 틀렸나} → {고친 내용} (근거: {확인한 출처})
[제안] {무엇이 틀렸나} → {제안하는 수정} (근거: {확인한 출처}) — 계획·미실행 대상일 때, 직접 안 고치고 제안만
[불확실] {증거로 확인 못 한 것} → {왜 / 어떻게 확인 가능}
[확인됨] {의심했지만 검토 결과 맞았던 핵심 주장} (선택, 1~2개만)
[다음] {지금 사용자가 할 행동} — 승인 필요 여부 / 반영 여부 / 진행 가능 여부를 한 줄로
```

- 수정이 답 자체를 바꾸면, 고친 최종 답을 다시 제시한다.
- **마지막 줄은 항상 `[다음]`** — 검토 결과를 사용자의 다음 행동으로 번역한다. 건수와 승인 필요 여부를 명시한다: 수정 없으면 "문제 없음 — 다음 작업 진행 권장", `[제안]`이 남으면 "제안 N건, 승인 필요 — '진행'이면 반영", `[수정]`을 이미 반영했으면 "수정 N건 반영 완료 — 확인 후 다음 단계로", `[불확실]`만 남으면 "불확실 N건 — 확인할지 결정 요망". 막연한 "검토 완료"는 금지(아래와 동일 기준).
- 증거 있는 수정이 하나도 안 나오면 "재검토 결과 근거 있는 수정 없음 — {가장 약했던 가정 1개와 그게 버틴 이유}"로 정리하고, `[다음] 문제 없음 — 다음 작업 진행 권장`으로 마무리한다. (막연한 "확인 완료" 금지.)
