# Pick Skill — 진입점 라우터

사용자가 계획/요청을 적으면, **진행 전에** 어떤 스킬 또는 모드로 가는 게 적합한지 골라 추천한다.
목적은 사용자가 매번 "이거 무슨 스킬 쓰지?"를 손으로 묻는 라운드를 대신하는 것이다.

규칙 12를 따른다 — 여기서는 **고르고 추천만** 한다. 실제 실행(스킬 호출/모드 진입)은 사용자가 고른 뒤 한다.

## 후보를 어떻게 모으나

후보는 두 곳에서 온다. **하드코딩한 카탈로그를 유지하지 마라** — 새 스킬이 추가되면 ①에 저절로 들어온다.

**① 라이브 스킬 목록 (자동 갱신)**
이 세션에 주입된 "사용 가능한 스킬 목록"을 본다. 사용자 스킬(`deep-interview`, `harness`, `self-review`, `draft`, `doc-sync` 등)과 플러그인 스킬이 모두 여기 있다. 각 스킬의 description이 곧 "언제 쓰나"이므로 그걸 매칭 근거로 쓴다.

**② 비-스킬 빌트인 미니목록 (수동 — 거의 안 늘어남)**
빌트인 슬래시 커맨드·모드는 ①에 안 잡히므로 여기서 직접 챙긴다:
- **plan mode** — 실행 전 엄격 차단이 필요한 다단계 구현. 전략을 먼저 합의하고 승인 게이트를 걸고 싶을 때.
- **/goal** — "완료 조건을 정해놓고 그게 충족될 때까지 멈추지 말고 자율 진행." 장기 실행 작업.
- **/loop** — 같은 작업을 주기적으로 반복(폴링·상태 감시).
- **/schedule** — cron 기반 예약 실행.

새 빌트인 커맨드를 알게 되면 이 목록에 한 줄 추가한다.

## 판단축

요청을 읽고 아래로 가장 잘 맞는 후보를 고른다:

- 요청이 **모호/범위 흐릿**, 목표·완료기준이 안 잡힘 → `deep-interview`
- **구현 전략을 먼저 합의하고 차단**하고 싶다(되돌리기 어려운 다단계) → **plan mode**
- **"완료 조건 정해놓고 끝까지 자율로"** → **/goal**
- **반복 자동화·에이전트 팀 체계** 구축 → `harness`
- 직전 답·산출물을 **적대적으로 재검토** → `self-review`
- KBS **기안문·보고서** 작성 → `draft`
- 특정 도메인 산출물(docx/xlsx/pptx/pdf/디자인 등) → 해당 `document-skills:*`
- 위 어디에도 안 맞고 **그냥 바로 하면 되는 단순 작업** → 추천 없이 직접 처리

## 진행 방식

1. 요청을 한 문장으로 요약한다.
2. 라이브 스킬 목록 + 빌트인 미니목록에서 매칭되는 후보를 추린다.
3. **1순위 1개 + 이유**를 제시한다. 근소하게 겹치는 차순위가 있으면 1개까지만 덧붙인다(나열 금지).
4. 자명하면 굳이 묻지 말고 추천만 하고, 사용자가 고르면 그 스킬/모드로 넘어간다.

## 출력 형식

```md
요약: {요청 한 문장}
추천: {스킬/모드 1개} — {왜 이게 맞나, 한 줄}
(차순위: {있으면 1개} — {언제 이쪽이 나은가})
다음: "{추천}으로 진행할까요?" / 사용자가 고르면 해당 스킬 호출·모드 진입
```

자명한 단순 작업이면 위 형식 생략하고 바로 처리한다.
