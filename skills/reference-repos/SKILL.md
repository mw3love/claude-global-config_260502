# reference-repos — 완성된 프로젝트에서 prior art 참고

새 작업에서 비자명한 설계·접근을 정하기 전에, **이미 비슷한 문제를 풀어 둔 내 git 프로젝트**가 있는지 찾아 그 해법을 근거로 삼는다. git에 올라간 repo는 어느 정도 완성된 것이 많아 참고 가치가 높다. 경로는 PC마다 다르거나 없을 수 있지만 **git remote는 항상 정확**하므로 remote를 키로 접근한다.

## 두 개의 저장소 — 위키(내용)와 인덱스(라우팅)

**1. `~/.claude/wiki/*.md` — 함정 위키. 내용은 전부 여기 있다.**
기법 하나 = 파일 하나. 파일이 분리돼 있어야 PC 간 병합이 충돌 없이 되고(같은 줄을 안 건드림), grep으로 항목 단위 검색이 된다. 담는 것은 **코드로 복원 불가능한 것만** — 막다른 길, 왜 실패했나, 최종 해법. "이 기능 어떻게 만들었나"는 절차 4에서 repo 코드를 직접 읽어 복원하므로 적지 않는다(코드는 절대 낡지 않지만 노트는 낡는다).

**2. `~/.claude/repos.json` — 어느 repo를 열지 고르는 라우팅 인덱스.** (sync-repos와 공용 파일이지만 sync는 `path`/`build`만 읽으므로 무관)

- `remote` — git URL. **PC 불문 안정 키.**
- `reference` — "이 repo가 잘 풀어 둔 문제" **한 줄 요약**. 라우팅용이지 내용 저장소가 아니다. 여기에 기법을 이어붙이지 않는다.
- `path`(선택) — 홈(`~`) 기준 상대경로. 이 PC에 로컬 클론이 있을 때의 위치 힌트.

`reference`가 있고 `path`가 없는 항목 = **참고 전용 repo**(sync-repos는 자동 제외함).

## 절차

0. **위키 먼저 grep** — `~/.claude/wiki/`에서 지금 문제의 키워드(기능명·API명·증상)를 찾는다. 함정 기록은 다른 어디서도 복원되지 않으므로 여기가 1순위다. 맞는 항목이 있으면 읽고, **참고했으면 그 파일 frontmatter의 `used:`에 한 줄 append**(아래 «사용 기록»).

1. **인덱스 로드** — `~/.claude/repos.json`을 읽어 `reference` 있는 항목만 추린다. 이건 "어느 repo의 코드를 열까"를 고르는 용도다.

2. **지목 우선 + 인덱스 매칭** — 가장 신뢰도 높은 신호는 **사용자가 이름으로 지목한 repo**다(만든 사람이 기능을 가장 잘 안다). 지목이 있으면 그걸 1순위로. 더해서, 지금 풀려는 문제를 각 `reference` 노트와 대조해 관련 repo를 고른다.

3. **라이브 GitHub 스캔(놓침 줍기·인덱스 보강)** — 인덱스 매칭이 비었거나 얕을 때, 또는 사용자가 "GitHub에서/내 프로젝트 중 참고할 거 훑어봐"처럼 요청하면, 소유자의 repo 목록을 실시간으로 받아 인덱스에 없던 prior art까지 포괄한다:
   - **username 추출**: `repos.json`의 remote(`github.com/<user>/...`)나 `git config`에서 뽑는다.
   - **공개 repo (토큰 불요)**: `curl -s "https://api.github.com/users/<user>/repos?per_page=100&sort=updated"` → 이름·설명·언어로 후보를 추린다. `gh` 미설치 PC에서도 동작.
   - **비공개 repo**: 익명 API엔 안 뜬다 → `gh`(설치 시)나 토큰이 있어야 보임. 없으면 "공개분만 훑음, 비공개는 직접 지목 요망"으로 알린다.
   - GitHub 설명이 비어 이름만으론 거칠면, 관련돼 보이는 후보를 **사용자에게 제시**("이것들 관련 있어 보임 — 볼까요?")하거나 README를 가볍게 열어 확인한다. 추측으로 단정하지 않는다.

4. **코드 확보(PC 독립)** — 고른 repo의 실제 코드를 연다:
   - **로컬 클론 우선**: `path`가 있으면 `~/<path>` 확인. 없거나 `path`가 비었으면, repo 이름으로 흔한 개발 폴더(`~/Dev`, `~/Documents` 등)에서 `.git` remote가 일치하는 클론을 가볍게 탐색.
   - **없으면 shallow clone 캐시**: `git clone --depth 1 <remote> ~/.claude/ref-cache/<repo-name>` 로 받아 읽는다. 이미 캐시에 있으면 재사용(필요 시 `git -C <cache> pull --ff-only`로 갱신). 캐시 디렉터리는 읽기 전용 참고용이며 수정하지 않는다.
   - clone 실패(네트워크·인증)면 그 사실을 보고하고 나머지 후보로 진행.

5. **해법 추출** — 관련 파일을 읽어(README/PLAN/HANDOFF 같은 설계문서가 있으면 먼저, 그다음 핵심 소스) **현재 문제에 옮길 수 있는 접근**을 요약한다. 그대로 베끼지 말고, 왜 그렇게 풀었는지·우리 상황과 다른 점·옮길 때 바꿔야 할 부분을 함께 제시한다.

6. **판단 보조 + 인덱스 자가성장** — 추출한 접근을 현재 작업의 선택지·추천에 **근거로** 연결한다(prior art가 정답이라는 보장은 없으니 맹신 말고 트레이드오프를 함께 평가). 라이브로 찾아 *실제로 참고한* repo가 인덱스에 없으면 아래 «기록(쓰기)» 절차로 자동 등록한다(미리 카탈로그화하지 않고 **사용에서 인덱스가 자라게**).

## 기록(쓰기) — 묻지 말고 자동, 한 줄 고지

읽기(위 절차)의 짝. `~/.claude/wiki/`에 새 파일을 **자동 생성**한다(CLAUDE.md 규칙 4-c). 사용자에게 묻지 않고, 한 줄만 고지: «📚 wiki 기록 — {기법 한 토막}». 사용자는 다음 턴에 취소할 수 있다.

### 언제 — 트리거는 하나뿐

**규칙 11-b 스턱루프가 성공으로 풀린 순간**(안 되다 안 되다 드디어 됨). *그 순간 바로* 기록한다(나중으로 미루면 '왜 어려웠나'가 증발).

**"재사용 가치 있어 보이는 비자명한 기법"은 기록하지 않는다.** 판단이 개입하는 품질바는 늘 통과되어 검색 5분짜리 일반지식이 쌓인다(실제로 그렇게 오염됐다). 고생했으면 들어가고, 안 했으면 안 들어간다. 애매하면 **안 넣는다.**

### 무엇을 — 파일 하나 = 기법 하나

`~/.claude/wiki/<repo>-<기법>.md`:

```markdown
---
repo: PasteFlow
remote: https://github.com/mw3love/Paste_flow.git
stack: [PyQt6, Windows]
tags: [영역캡처, 오버레이, MSAA, DWM]
used: []
---

# 제목

## 함정
무엇을 시도했고 왜 안 됐나. **값은 여기 있다.**

## 해법
최종적으로 통한 것.

## 대가
그 해법이 새로 만든 문제 (있으면).
```

- **막다른 길·왜 그게 안 됐나·최종 해법**을 쓴다. 결과만 적지 말 것.
- **기능 설명·API 사용법은 쓰지 않는다.** 절차 4에서 코드를 읽으면 나오고, 노트보다 코드가 항상 최신이다.
- `tags`는 grep 매칭용 키워드 — 기능·함정 이름을 충분히.
- 기존 파일에 같은 함정이 있으면 새로 만들지 말고 보강한다(dedup).

### 사용 기록 (`used:`) — 위키의 존재 이유를 측정하는 유일한 장치

절차 0에서 항목을 **실제로 참고했으면** 그 파일 frontmatter에 한 줄 append:

```yaml
used:
  - 2026-09-14 AI-CAD (회전 핸들 잔상)
```

쓰기는 커밋에 흔적이 남지만 읽기는 아무것도 안 남는다. 이 기록이 없으면 "이 위키가 쓸모 있나"를 영원히 답할 수 없고, 관측되지 않는 산출물은 의식(ritual)이 된다. **오래도록 `used:`가 빈 항목은 삭제 후보다.**

### 라우팅 인덱스 갱신 (새 repo일 때만)

그 repo가 `repos.json`에 아예 없을 때만 한 줄 추가한다. 기존 repo의 `reference`에 기법을 이어붙이지 않는다.

```json
{ "desc": "프로젝트 설명 [참고 전용]", "remote": "https://github.com/.../repo.git",
  "reference": "이 repo가 잘 풀어 둔 문제 — 한 줄 요약" }
```

### 커밋 — 자동, push는 하지 않음

`~/.claude`는 지금 작업 중인 프로젝트와 **다른 저장소**라, 그 프로젝트를 push해도 여기 변경은 영원히 커밋되지 않는다. 그래서 기록 직후 바로 커밋한다.

```bash
git -C ~/.claude add -N wiki/<파일>.md                        # 새 파일은 의도만 등록
git -C ~/.claude commit wiki/<파일>.md -m "wiki: <기법 한 토막>"
```

- **`add -N` 이 빠지면 안 된다.** 새 파일은 untracked라 pathspec 커밋이 `pathspec did not match any file(s) known to git` 으로 **실패한다**(실측). `add -N`은 "이 파일을 추적할 의도"만 등록하고 내용은 스테이징하지 않으므로, 이어지는 pathspec 커밋의 격리가 유지된다.
- **반드시 pathspec 형태**(`commit <경로>`)를 쓴다. `git add … && git commit` 은 인덱스 전체를 커밋해 **사용자가 스테이징해 둔 다른 작업까지 삼킨다**(실측). `-A`는 절대 금지.
- **push하지 않는다.** 자동 push는 다른 PC와의 rebase 충돌을 부른다. 다음에 `~/.claude`를 push할 때 함께 올라간다.
- 커밋 실패(index.lock 경합 등)는 조용히 넘기고 다음 기회에 다시 시도한다. 파일은 이미 디스크에 있으므로 기록이 사라지지 않는다.

## 안전 가드

- 캐시 클론(`~/.claude/ref-cache/**`)과 기존 로컬 클론은 **읽기 전용 참고용** — 수정·커밋·push 하지 않는다. (`ref-cache/`는 `.gitignore` 대상 — 중첩 `.git`이 있어 `git add -A` 하면 경고만 내고 gitlink로 박힌다.)
- `wiki/` 기록은 위 «기록» 트리거(스턱루프) 하에서 **자동**이다(그 외 임의 편집은 사용자 요청 시). 코드 파일·다른 repo는 건드리지 않는다.
- git mutating 명령은 **딱 하나만** 허용: `git -C ~/.claude commit <경로> -m …` (pathspec 형태, push 없음). 그 외 `add -A`·`push`·`reset`·`checkout`·`rebase` 금지.
