로그온 자동실행처럼 사람이 지켜보지 않는 백그라운드 스크립트의 완료 신호를, 처음엔 "문제 있을 때만 알림(노이즈 최소화)"으로 설계했으나 사용자가 "항상 알림 뜨게 하고 pull할 게 없으면 없다고 알림"으로 뒤집었다.

**왜:** "조용함=성공"설계는 두 경우(⓵ 정말 변경없음 ⓶ 스크립트 자체가 실행조차 안 됨)를 똑같이 "알림 없음"으로 만들어 사용자가 구별 못 한다는 게 사용자의 지적(정확했음). "항상 알림"으로 바꾸면 매 실행마다 정확히 1개 뜬다는 기대가 생기고, 그 기대가 깨지는 것(알림 자체가 없음) 자체가 "자동화가 안 돌았다"는 신호가 된다 — 관측 빈도가 낮은(로그온 1일 1회 등) 자동화에서는 "노이즈"가 아니라 관측성 이득이 더 크다.

**적용 시점:** 백그라운드/자동화 작업의 완료 알림을 설계할 때. 내 기본 성향(노이즈 최소화 → 실패시에만 알림)을 먼저 제안하되, 실행 빈도가 낮고(하루 1회 이하 등) "무소식이 진짜 성공인지 실행 자체가 안 된 건지" 구별이 중요한 경우엔 "항상 알림"을 대안으로 먼저 언급한다. 구현 시엔 전체 로직을 넓은 try/except로 감싸 코드 내부 예외도 최소 알림 한 줄은 뜨게 해야 이 설계의 효과가 완성된다(그렇지 않으면 "항상 알림"도 스크립트 크래시 앞에서는 똑같이 무력함).

**손안의 카드 확인 누락 사례(더 중요한 교훈):** 알림을 구현하며 규칙 2("이미 있는 시스템이 제공하는가 먼저 확인")를 건너뛰고 WinRT 토스트를 직접 새로 짰다. 그런데 `~/.claude`에는 **이미 훨씬 나은 알림 디스패처(`toast.sh`→`toast.ps1`)가 있었다** — Windows 토스트+화면중앙 팝업(`center-toast.ps1`)+Telegram 발송까지 갖춘, Claude 답변완료/질문대기 알림용으로 이미 쓰이던 시스템. `README.md`를 doc-sync로 훑다가 뒤늦게 발견해 자체 구현을 걷어내고 이걸로 교체했다. **다음에 "이 PC/이 리포에 알림 수단이 있나"를 물을 상황이면 새로 만들기 전에 `~/.claude/toast.sh`부터 확인할 것.**

**부수 정정(도구 신뢰성):** WinRT 토스트가 실측에서 예외를 던진 원인은 "이 PC가 WinRT를 못 씀"이 아니라 **내가 만든 호출에 `-Sta`(단일 스레드 아파트먼트) 플래그가 빠져서**였다(WinRT COM은 STA 필수). 기존 `toast.ps1`은 `-NoProfile -Sta -ExecutionPolicy Bypass`로 정상 호출해 이 PC에서도 WinRT 토스트가 잘 뜬다. 즉 "이 PC에서 WinRT 토스트는 안 됨"이라는 이전 기록은 **오판이었다** — 손안의 카드를 먼저 확인했다면 이 오판 자체가 없었을 것(기존 스크립트를 봤다면 `-Sta`가 필요하다는 걸 바로 알았을 것).
