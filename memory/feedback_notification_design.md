로그온 자동실행처럼 사람이 지켜보지 않는 백그라운드 스크립트의 완료 신호를, 처음엔 "문제 있을 때만 알림(노이즈 최소화)"으로 설계했으나 사용자가 "항상 알림 뜨게 하고 pull할 게 없으면 없다고 알림"으로 뒤집었다.

**왜:** "조용함=성공"설계는 두 경우(⓵ 정말 변경없음 ⓶ 스크립트 자체가 실행조차 안 됨)를 똑같이 "알림 없음"으로 만들어 사용자가 구별 못 한다는 게 사용자의 지적(정확했음). "항상 알림"으로 바꾸면 매 실행마다 정확히 1개 뜬다는 기대가 생기고, 그 기대가 깨지는 것(알림 자체가 없음) 자체가 "자동화가 안 돌았다"는 신호가 된다 — 관측 빈도가 낮은(로그온 1일 1회 등) 자동화에서는 "노이즈"가 아니라 관측성 이득이 더 크다.

**적용 시점:** 백그라운드/자동화 작업의 완료 알림을 설계할 때. 내 기본 성향(노이즈 최소화 → 실패시에만 알림)을 먼저 제안하되, 실행 빈도가 낮고(하루 1회 이하 등) "무소식이 진짜 성공인지 실행 자체가 안 된 건지" 구별이 중요한 경우엔 "항상 알림"을 대안으로 먼저 언급한다. 구현 시엔 전체 로직을 넓은 try/except로 감싸 코드 내부 예외도 최소 알림 한 줄은 뜨게 해야 이 설계의 효과가 완성된다(그렇지 않으면 "항상 알림"도 스크립트 크래시 앞에서는 똑같이 무력함).

**부수 발견(도구 신뢰성):** Windows에서 완료 알림 구현 시 WinRT 토스트(`Windows.UI.Notifications.ToastNotificationManager`)는 이 사용자의 PC(MW-Lenovo, Windows 11 Pro)에서 실측 실패(타입 로드 실패·컬렉션 열거 예외, PowerShell 7/5.1 둘 다)했다. `System.Windows.Forms.NotifyIcon` 풍선말풍선(구식 .NET, 별도 설치 불요)으로 대체해 실측 성공. 향후 이 PC에서 "손안의 카드"로 WinRT 토스트를 가정하지 말 것 — 확인 없이 새로 추천하면 또 실패한다.
