# OpenAI 호환 LLM 게이트웨이 — 모델 능력 전수 스윕

## 함정 — `/models`가 광고하는 모델이 실제로는 안 된다

OpenAI 호환 게이트웨이의 `/models` 목록을 믿으면 두 종류 지뢰를 밟는다.

- **유령 모델** — 목록엔 있는데 호출하면 **404**
- **텍스트 전용 모델** — 텍스트는 되는데 이미지 입력을 주면 **400**

목록만으로는 구분이 안 된다. OCR(비전 필수) 기능에 텍스트 전용 모델을 물리면 런타임에 터진다.

## 해법 — 단색 PNG로 능력을 직접 때려본다

- **단색 PNG 색 판별**로 이미지 입력 가능 여부를 실측한다.
- **빨강/파랑 교차 검증**으로 위양성(우연히 맞힌 것)을 제거한다.
- 결과로 **OCR용 모델 설정과 AI 질의용 모델 설정을 분리**한다.

## 함정 속 함정 — 자동 폴백이 실패를 성공처럼 감춘다

`_call_with_fallback` 같은 자동 폴백이 있으면, **유령 모델 호출 실패가 폴백 성공으로 덮여** 스윕 결과가 거짓 양성이 된다.

**해법:** `last_used_model != 요청 모델` 로 걸러낸다. 폴백이 다른 모델로 답했으면 원래 요청 모델은 실패한 것으로 친다.

## 인접 함정 — Gemini grounding 폴백

Gemini 웹검색 grounding(`google_search` 도구)을 쓸 때, **검색 무료 할당량이 없는 모델**은 `429`를 뱉는다. → 안전망 모델로 자동 폴백해야 한다. (SDK는 폐기된 `google-generativeai` → 신 `google-genai` 로 이전.)

## 함정 1 — 무작위 영숫자를 저대비로 그리면 CAPTCHA가 된다

작은 글씨·저대비 배경 위에 **맥락 없는 무작위 영숫자**를 뿌려 "이걸 읽어라" 하면, 안전 정렬된
모델(claude 계열)이 `finish_reason=content_filter` + 빈 본문으로 **거부**한다. CAPTCHA 풀이
요청과 구별되지 않기 때문. **OCR 능력이 아니라 거부 정책을 재고 있었다** — 정렬이 잘 된 좋은
모델일수록 탈락하는 역선택.

**해법:** 토큰을 자연스러운 문서 안에 심는다(로그 문장 `[10:24:12] 커밋 A87A 배포 대기열에…`,
컬럼명 있는 표). 무작위성(추측 방지)은 유지하면서 스크린샷으로 읽히게 한다. 고대비·큰 글씨
이미지는 거부되지 않으므로, **거부는 난이도 올린 이미지에서만 나타나 원인 파악이 늦다.**

## 함정 2 — 한글 무작위 낱글자는 통째로 최소대립쌍

`"가나다라마바사아자차카타파하거너더러머버서어저처커터퍼허"`에서 낱글자를 뽑아 토큰을 만들었다.
이 알파벳은 **자음마다 ㅏ/ㅓ 최소대립쌍**(가/거, 라/러, 하/허 …)이다. 12px에서 ㅏ와 ㅓ는 획 하나
방향 차이라 사람도 구별 못 한다. 정답 `카러차`를 거의 모든 모델이 `카라차`로 읽어 실패했다.

ASCII 토큰에서는 `0/O`, `1/I`를 일부러 뺐으면서 **한글 알파벳은 혼동쌍만으로 구성**한 것.
같은 원칙을 한쪽에만 적용했다는 걸 데이터가 나올 때까지 못 봤다.

**해법:** 낱글자 조합 대신 **실제 단어 목록에서 무작위 선택**(`하늘빛`, `겨울나무`, `종이배`…).
어휘 문맥이 있어 실사용 조건에 가깝고, 선택이 무작위라 예측도 불가능하다.

## 근본 해법 — 완전일치를 버리고 recall로

두 번 고치고도 문제가 남은 이유는 **채점 메커니즘 자체**였다. "무작위 토큰을 전부 맞혔나"는
**글자 하나가 판정을 통째로 뒤집는다.** 최악 케이스 문자 변별력을 재는 것이지 "실사용 텍스트를
읽는가"를 재는 게 아니다.

→ **정답 전문 대비 복원율(recall)**: `sum(matching_blocks) / len(truth_normalized)`
(`difflib.SequenceMatcher`, 공백 제거 + NFC + 대문자화 후). 임계 **0.95**.

- 글자 1개 미끄러짐 = 0.99 (통과), 문장 붕괴 = 0.84 (탈락)
- matching block은 순서를 보존하므로 모델이 `"이미지의 텍스트는:"` 같은 **서두를 붙여도 recall 불변**
  (분모가 정답 길이라 여분 출력이 점수를 올리지도 내리지도 않음)
- 45종 실측 분포에 **0.94와 0.96 사이 자연스러운 빈틈**이 있어 임계값이 자의적이지 않았다

**남는 한계:** recall은 *오류의 종류*를 구분하지 못한다. 타임스탬프 콜론을 마침표로 4번 읽은 모델과
식별자 `BA9E`를 `BAGE`로 읽은 모델이 **똑같이 0.941**이 나온다. 가중치를 주려면 임의 판단이 들어간다.

## 부수 교훈 (전부 실제로 비용을 치렀다)

- **원시 덤프에 정답 텍스트와 seed를 저장하지 않으면 재채점이 불가능하다.** 모델 출력만 있고 무엇과
  비교했는지가 없어 임계값 조정에 API 비용을 다시 썼다. → `truth`·`seed`·`recall`을 함께 덤프하고,
  **덤프만으로 재분류하는 `--from-raw` 모드**를 둔다. 분류기 버그를 고쳤을 때 돈을 안 쓰고 재빌드된다.
- **`fail`과 `unknown`은 다르다.** 429/503을 "모델이 못 함"으로 굳히면 UI가 멀쩡한 모델을 영구 회색
  처리한다(공식 API 30종 중 21종이 429였다). "못 한다"와 "못 재봤다"를 반드시 분리할 것.
  레벨을 *한 번도 시도하지 않은* 경우도 `fail`이 아니라 `unknown`이다.
- **오류 분류기는 일시적 오류를 맨 먼저 본다.** `not_found`를 먼저, 그것도 맨 문자열 `"404"` 포함으로
  판정하면 **429 메시지가 유령 모델로 오분류**된다(실제로 발생). 순서: 429 → 503 → timeout →
  엔드포인트 미지원 → 404 → 이미지 400.
- **"유령 모델"이 사실은 유령이 아닐 수 있다.** 게이트웨이의 codex 계열 404 본문은
  `"not supported in the v1/chat/completions endpoint. Use the v1/responses endpoint"` —
  모델은 존재하고 **엔드포인트가 다를 뿐**이다. UI에서 막는 결론은 같아도 사용자에게 보여줄 이유가 다르다.
- **응답의 `model` 필드를 요청 모델과 대조**하면 게이트웨이의 무단 대체를 잡을 수 있다. 단
  `gpt-5-mini` → `gpt-5-mini-2025-08-07`처럼 **날짜 스냅샷 별칭은 정상**이므로
  `served.startswith(model + "-")`를 별칭으로 통과시켜야 헛경보가 안 난다.
- **스모크 3종을 먼저 돌린다.** 위 함정 1·2는 전수 스윕 전에 3종만 돌려 잡았거나(1), 못 잡아서 전수
  스윕 한 번을 통째로 버렸다(2). 전수 스윕은 비싸다.
